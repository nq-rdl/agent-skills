# Scripts Language Guide

Skills may ship a `scripts/` directory containing executable tooling invoked by the agent. This document defines the supported languages, their conventions, and platform assumptions.

## Platform Assumptions

Agent skills assume **macOS, Linux, or Linux via WSL2**. Windows (native, non-WSL) is not supported. Scripts may use POSIX-compatible shells, standard Unix utilities, and language runtimes available on these platforms.

## Supported Languages

### Bash

Use for lightweight glue scripts, environment setup, and simple automation.

- Shebang: `#!/usr/bin/env bash`
- Use `set -euo pipefail` at the top of every script
- Prefer POSIX-compatible constructs; avoid bashisms that differ between macOS and Linux
- Not suitable for complex logic — use Python, TypeScript, or Go instead

### Python

Use for data processing, PDF/document handling, CLI tools, and anything needing the PyPI ecosystem.

- Minimum version: **Python 3.13+**
- Dependency management: `requirements.txt` + `ensure-deps.sh` (auto-detects pixi > uv > mamba > conda > pip)
  - End-users should prefer `uv` for installing skill deps: `uv pip install -r requirements.txt`
- CLI framework: `typer` (preferred) or `click`
- Do not commit virtual environments or `.pixi/` directories
- Dev loop: `pixi run -e <skill> test` / `pixi run -e <skill> typecheck` (see per-skill pixi features in `pyproject.toml`)

### TypeScript (Bun)

Use for MCP servers, Claude SDK integrations, and scripts that benefit from the npm ecosystem.

- Runtime: **Bun 1.0+** (not Node.js — Bun is the standard runtime for TS skills)
- Dependency management: `package.json` + `bun install`
- Test runner: `bun:test`
- Do not commit `node_modules/`

> **Note on MCP servers:** Go is now the preferred language for new MCP servers in this repo (see below). TypeScript MCP servers are still supported but the Go pattern is canonical.

### Go

Use for long-running services, ConnectRPC/gRPC servers, CLI binaries, MCP servers, and performance-sensitive tooling.

- Minimum version: **Go 1.25+** (pinned in `/.go-version`)
- Each skill with Go scripts has its own `go.mod` inside `scripts/` — no shared module
- The root `go.work` unifies all Go modules for dev (`go test ./...`, IDE support)
- Build tool: standard `go build` + `Makefile` (no pixi required)
- Code generation (protobuf): use `buf` with a `buf.gen.yaml`; commit generated code under `gen/`
- Do not commit `bin/` directories or compiled binaries
- Release binaries are distributed via GitHub Releases (goreleaser); see `.goreleaser.yaml`

#### Adding a Go MCP server to a skill

To expose a skill's logic as an MCP server (stdio transport):

1. Add `cmd/<skill-name>-mcp/main.go` inside the skill's `scripts/` directory.
2. Import `github.com/mark3labs/mcp-go` — the community Go MCP SDK.
3. Follow the `pi-mcp` pattern in `skills/pi-rpc/scripts/cmd/pi-mcp/`:
   - Create an `mcp.Server`, register one tool per action (`session_create`, `session_list`, etc.)
   - Call `server.ServeStdio(s)` — the agent-extensions launcher handles process lifetime
4. Add a `build-mcp` target to the skill's `Makefile`.
5. Add the binary to `.goreleaser.yaml` under `builds:`.

Example skeleton:

```go
package main

import (
    "context"
    "encoding/json"
    "log"
    "os"

    "github.com/mark3labs/mcp-go/mcp"
    "github.com/mark3labs/mcp-go/server"
)

var version = "dev"

func main() {
    s := server.NewMCPServer("my-skill-mcp", version,
        server.WithToolCapabilities(true))

    s.AddTool(
        mcp.NewTool("my_action",
            mcp.WithDescription("Does something useful"),
            mcp.WithString("param", mcp.Required(), mcp.Description("Input parameter")),
        ),
        func(ctx context.Context, req mcp.CallToolRequest) (*mcp.CallToolResult, error) {
            param := req.Params.Arguments["param"].(string)
            result := map[string]string{"result": param}
            data, _ := json.Marshal(result)
            return mcp.NewToolResultText(string(data)), nil
        },
    )

    if err := server.ServeStdio(s); err != nil {
        log.Fatal(err)
    }
}
```

## Choosing a Language

| Situation | Preferred Language |
|-----------|-------------------|
| Simple file operations, env setup | Bash |
| Data extraction, document processing, PyPI dependency | Python |
| MCP server, Claude SDK wrapper, npm ecosystem | Go (preferred) or TypeScript (Bun) |
| Long-running service, ConnectRPC, CLI binary | Go |

## No Windows-specific code

Do not use:
- Windows path separators (`\`)
- `cmd.exe` or PowerShell syntax
- Windows-only APIs or registry access
- `.bat` or `.ps1` scripts

If a script needs to handle both macOS and Linux differences (e.g. `sed -i` syntax), use a compatibility shim or prefer Python/Go for that logic.
